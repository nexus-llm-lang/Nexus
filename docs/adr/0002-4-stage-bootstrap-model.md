# ADR 0002 — 4-Stage Bootstrap Model (stage0 / stage1 / stage2 + seed identity)

**Status**: Accepted  
**Date**: 2026-05-08  
**Issues**: nexus-dvr6.2, nexus-dvr6.3, nexus-dvr6.8  

## Context

When nexus transitioned from a Rust-hosted bootstrap to a fully self-hosting
compiler, the question of how to manage the compiler's own build chain
became concrete.  The Rust bootstrap (`bootstrap/`) provided an escape hatch:
the compiler's source could always fall back to a known-correct Rust
implementation to compile itself.  That escape hatch has a cost — the Rust
toolchain is a permanent build dependency, the codegen paths diverge, and
bugs in the Rust bootstrap can mask bugs in the nexus self-host.

### The bootstrap problem

A self-hosting compiler must compile itself.  The first compiler that compiles
the source must come from somewhere — this is the *seed*.  Three properties
are desirable:

1. **Reproducibility**: anyone with the same source tree gets the same output.
2. **Verifiability**: the seed can be checked against a freshly-built
   compiler; discrepancies surface tampering or drift.
3. **No external toolchain**: the build should not require Rust (or any other
   language) once the seed is committed.

A naïve single-stage bootstrap (`seed → final`) is fragile: if the seed
produces a compiler that compiles itself differently than the seed does, the
output is non-deterministic across seed versions.

### Options considered

| # | Option | Summary |
|---|--------|---------|
| 1 | Keep Rust bootstrap permanently | Stage0 = Rust, stage1 = nexus self-host |
| 2 | Committed seed + 2-stage compile | seed → stage1 → stage2 (=stage1 identity check) |
| 3 | 4-stage model with explicit identity | stage0(seed) → stage1 → stage2 → stage3 (identity at stage2==stage3) |
| 4 | No staged model; always rebuild from source | Requires trusting external toolchain |

## Decision

**Option 2 extended to 4 logical stages with an identity check at stage2.**

The model that landed (nexus-dvr6.2 through dvr6.8):

- **Stage 0**: the *committed seed* (`nexus.wasm` tracked in git) compiles
  `src/` to produce `stage1.wasm`.  The seed is the only binary that is
  not compiled from nexus source during a normal build.
- **Stage 1**: `stage1.wasm` compiles `src/` again to produce `stage2.wasm`.
- **Stage 2 / identity check**: `stage2.wasm` must byte-for-byte equal
  `stage1.wasm`.  This is verified by `bootstrap.sh --ci` (nexus-dvr6.3,
  commit `12519102`).  A mismatch means the seed is stale or the compiler
  has a non-deterministic codegen path.
- **Seed policy**: any `src/**` or `nxlib/**` change that affects codegen
  must commit a regenerated `nexus.wasm`.  The committed seed is therefore
  always the output of the previous fixpoint compile.

The Rust `bootstrap/` directory was deleted at nexus-dvr6.8 (commit
`753d5c2e`).  CI now uses the committed seed exclusively.

A fourth "stage" is implicit: `nexus.wasm` is regenerated by the developer
and committed, becoming the new seed for the next cycle.  This closes the
bootstrap loop without external tooling.

### Rationale

- The 3-stage identity check is the industry-standard technique for
  detecting compiler quining bugs (Trusting Trust problem).  Two stages
  would not catch a compiler that codegen-differs on its second pass.
- Committing `nexus.wasm` as a tracked binary makes `git clone` + `wasmtime`
  sufficient to build nexus without any other compiler.  The seed is auditable
  via `git log nexus.wasm`.
- Deleting `bootstrap/` removes the Rust dependency from the critical path
  and forces all codegen correctness to be verified through the nexus
  self-host path.

## Consequences

- **Positive**: the build chain has zero external compiler dependencies after
  the initial seed is accepted.
- **Positive**: `bootstrap.sh --ci` provides a regression gate for
  non-deterministic codegen; fixpoint failures are caught automatically.
- **Positive**: seed provenance is in git history; every `nexus.wasm` commit
  is auditable.
- **Negative**: any `src/` change that touches codegen requires a manual seed
  regeneration and commit.  Forgetting to regenerate breaks Stage 0 for
  downstream developers.
- **Negative**: Stage 0 can fail with cryptic NUL-byte symbol errors if the
  committed seed predates a symbol rename (see bootstrap-env-gotchas memory).
  Mitigation: always rebase worktrees onto the latest `main` before doing
  meaningful codegen work.
- **Neutral**: `nexus.wasm` binary diffs are noisy in `git log`.  Marked
  as binary in `.gitattributes` to suppress the diff noise.
